=== Afspraken

* **Backups & herstelprocedures**

De CRM-omgeving maakt gebruik van een centrale backupstrategie op applicatieniveau.

De backup bestaat uit:

* een gecomprimeerde MySQL-dump (`.sql.gz`);
* een Borg-back-up van:
** de volledige applicatiemap: `/home/apps/storage/crm`
** de databaseback-ups in `/opt/backup`

De database wordt iedere nacht eerst als `.sql.gz`-bestand weggeschreven naar `/opt/backup`. Vervolgens worden zowel deze databaseback-ups als de applicatiebestanden opgenomen in dezelfde Borg-back-up. Hierdoor zijn database en bestanden afkomstig uit hetzelfde back-upmoment en kunnen ze consistent worden hersteld.

* Backups worden dagelijks uitgevoerd.
* Er wordt gebruikgemaakt van incrementele Borg-back-ups, waarbij alleen gewijzigde data sinds de vorige back-up wordt opgeslagen.
* De retentieperiode bedraagt maximaal **7 dagen** (rolling window).
* Herstel kan plaatsvinden van:
** de volledige applicatie (`/home/apps/storage/crm`);
** de database;
** afzonderlijke bestanden of mappen;
** een specifieke back-up (snapshot) binnen de retentieperiode.

Deze aanpak zorgt ervoor dat applicatiebestanden, documenten en database altijd vanuit hetzelfde back-upmoment kunnen worden hersteld.

==== Controleren of de nachtelijke back-up de database bevat

Controleer op de betreffende server in `/opt/backup`:

* de logbestanden van het back-upproces op fouten;
* of er `.sql.gz`-bestanden aanwezig zijn met een datum/timestamp van de afgelopen nacht.

Ontbreken deze bestanden of hebben zij een oude datum, dan is de databaseback-up niet aangemaakt. Los dit eerst op voordat een restore vanuit Borg wordt uitgevoerd.

Controleer vervolgens of ook de Borg-back-up van dezelfde nacht aanwezig is:

[source,bash]
----
borg list privatescan@10.0.0.1:~/data.privatescan.nl/borg/
----

==== De Borg-back-up bekijken

NOTE: De productieomgeving gebruikt **BorgBackup 1.2**.

Documentatie:
https://borgbackup.readthedocs.io/en/stable/index.html

Vereiste: log in als gebruiker `root`, omdat de SSH-toegang naar de back-upserver onder deze gebruiker is geconfigureerd.

Beschikbare back-ups opvragen van de productieomgeving:

[source,bash]
----
borg list privatescan@10.0.0.1:~/data.privatescan.nl/borg/
----

Beschikbare back-ups opvragen van de acceptatie-/dataomgeving:

[source,bash]
----
borg list privatescan@10.0.0.1:~/intern.privatescan.nl/borg/
----

De output toont alle beschikbare snapshots binnen de retentieperiode.

==== Een back-up terugzetten

. Bepaal met `borg list` welk archief moet worden hersteld.
. Maak een tijdelijke restorelocatie aan:
+
[source,bash]
----
mkdir -p /tmp/restore
cd /tmp/restore
----
. Herstel de databaseback-ups uit het gewenste archief:
+
[source,bash]
----
borg extract privatescan@10.0.0.1:~/<omgeving>/borg/::<archiefnaam> opt/backup
----
+
Vervang:
+
* `<omgeving>` door bijvoorbeeld `data.privatescan.nl` of `intern.privatescan.nl`;
* `<archiefnaam>` door de gewenste snapshot uit `borg list`.

. Controleer of de gewenste databaseback-up aanwezig is:
+
[source,bash]
----
ls -lh /tmp/restore/opt/backup
----

. Zet de database terug:
+
[source,bash]
----
gunzip -c /tmp/restore/opt/backup/<database>.sql.gz | mysql -u <user> -p <database>
----

. Controleer na de restore of de applicatie correct functioneert.

TIP: Wanneer ook applicatiebestanden moeten worden hersteld, extraheer dan de map `/home/apps/storage/crm` uit dezelfde Borg-snapshot. Hierdoor blijven database en bestanden synchroon.

=== Bekende Issues & Oplossingen

==== Log schrijfrechten probleem (OPGELOST)

**Symptoom:**
Na 2-3 dagen krijgen alle acties een foutmelding:

----
could not append log.. invalid rights.
Kon niet overnemen: The stream or file "/usr/share/nginx/html/storage/logs/laravel-2025-09-29.log"
could not be opened in append mode: Failed to open stream: Permission denied
----

**Oorzaak:**
Laravel's daily log rotatie maakt dagelijks nieuwe logbestanden aan (bijv. `laravel-2025-09-29.log`). Het probleem ontstond doordat:

1. PHP-FPM draaide als root gebruiker (via `-R` flag) in plaats van `www-data`
2. De `storage/logs` directory had geen correcte permissies voor `www-data` gebruiker
3. Bij het aanmaken van een nieuw dagelijks logbestand faalde dit door missende schrijfrechten

**Oplossing:**
De volgende aanpassingen zijn doorgevoerd:

1. **PHP-FPM configuratie** (`.docker/php/fpm-cloud.conf`):
   - Toegevoegd: `user = www-data` en `group = www-data`
   - Socket ownership aangepast naar `www-data:www-data`
   - Socket mode ingesteld op `0660`

2. **Supervisor configuratie** (`.docker/supervisord.conf`):
   - Verwijderd: `-R` flag van php-fpm command
   - Toegevoegd: `user=www-data` directive voor php-fpm programma

3. **Docker startup script** (`.docker/start.sh`):
   - Toegevoegd: Automatische permissie fix bij elke container start
   - Commando's:
     ```bash
     chown -R www-data:www-data /usr/share/nginx/html/storage
     chmod -R 775 /usr/share/nginx/html/storage
     chmod -R 775 /usr/share/nginx/html/storage/logs
     chmod -R 775 /usr/share/nginx/html/bootstrap/cache
     ```

4. **Dockerfile** (`docker/php/Dockerfile`):
   - Toegevoegd: Expliciete aanmaak van storage directories met juiste permissies
   - Directory ownership en permissions worden ingesteld tijdens image build

**Preventie:**
Deze fix zorgt ervoor dat de permissies bij elke container herstart automatisch worden hersteld, waardoor het probleem niet meer kan optreden.
